# CW-04: JavaScript - React — Reflection Responses

## 1. Component Design & Reusability

Splitting the UI into `HelloWorld`, `Counter`, `List`, and `FilteredList` meant each file only had to do one job, so I could build and test them one at a time instead of juggling everything at once. `List` is a good example of reusability — it just takes an `items` array as a prop and renders it, so it doesn't know or care whether those items are fruits and vegetables, todos, or products. I could drop it into a completely different app and it would still work as long as I pass it items with a `name`. `Counter` is another one I could reuse anywhere I need a standalone click counter — it manages its own `count` state internally, so nothing else has to know how it works. This separation also made debugging easier: when the search box wasn't filtering correctly, I only had to look inside `FilteredList`, not worry that the bug could be hiding in `App.js` or `List.jsx`.

## 2. State vs Props in UI Management

Props are the read-only data a parent hands down to a child — like `App` passing the `items` array into `FilteredList`, or `FilteredList` passing its already-filtered array into `List`. Those components can read `this.props.items` but they're not allowed to change it. State is different — it's data a component owns and manages itself, like `count` inside `Counter` or `search`/`type` inside `FilteredList`. I have to use `this.setState()` instead of writing `this.state.count = this.state.count + 1` directly because React only knows to re-render the component when `setState()` is called. Mutating `this.state` directly changes the value in memory, but React never finds out, so the screen would keep showing the old count even though the underlying data changed. `setState()` is what actually tells React "something changed, please re-render."

## 3. Asset Organization & Scalability

Right now everything fits in one `components/` folder because there are only four components, but at 50 components that would turn into an unreadable list. I'd switch to grouping by feature instead of dumping every file in one flat folder — something like `components/ui/` for small reusable pieces (buttons, inputs, dropdowns) and `components/features/` for bigger feature-specific components (like `FilteredList` and its supporting pieces). I'd keep naming consistent: PascalCase for component files (`FilteredList.jsx`) and camelCase for any helper/utility functions (`filterItems.js`). Each component would keep its matching CSS file next to it (`Counter.jsx` + `Counter.css`) instead of dumping every style into one giant `App.css`, which is already something I'd want to change even at this small scale.

## 4. DOM vs React's Virtual DOM

With plain JavaScript, updating the counter would mean grabbing the element with `document.getElementById('count')` and manually setting its `textContent` every single time it changes — and I'd have to do that by hand for every piece of the UI that can change. React instead lets me just describe what the UI should look like for a given state (`<h2>Current Count: {this.state.count}</h2>`) and it figures out the DOM updates itself. Behind the scenes, React keeps a lightweight in-memory copy of the DOM, and when `setState()` runs it compares the new version against the old one (diffing) and only touches the actual DOM nodes that changed (reconciliation) — so clicking Increment doesn't repaint the whole page, just the text node showing the count. That's both faster than manual DOM manipulation and easier to reason about, since I never have to track "did I remember to update this element" myself.

## 5. UI/UX Decisions for Learning & Accessibility

The search input and dropdown in `FilteredList` work fine with a mouse, but I noticed the react-bootstrap `DropdownButton` needs proper keyboard support (tab to focus, enter/space to open, arrow keys to move between options) and an `aria-label` so a screen reader announces it as a filter control rather than just an unlabeled button. I'd also add a visible count like "Showing 3 of 6 items" so users get feedback that a filter is actually applied, instead of just seeing the list silently shrink. For someone brand new to React, I think the hardest part isn't the JSX syntax — it's the mental model of the Virtual DOM, because it's invisible. You never see it, you just have to trust that React is doing the diffing correctly, which is a strange thing to accept on faith the first time you learn it. Pairing the abstract explanation with something concrete — like watching the count update in DevTools while only that one `<h2>` re-renders — is what made it click for me.
